## 2. 訂正: 第1報の 07-04 の行は部分日スナップショットだった [高]

第1報で「7/4 に委譲が 27% へ増えた」「cacheRd% は 95→93→92 と微減」と書いた観察は、

教訓: 測定日当日のバケットは確定値として扱ってはならない(本報の 07-05 も同様に部分日)。

### 3b. 正規化指標(第2報の本命)

定義: tok/req = 日次総tok/日次req。sessMed cRd/out = セッション単位の
cache_read/output 比の中央値(**req≥10 のセッションのみ**)。medReq/sess = 全セッションの
req 数中央値。hot率 = hot(≥80req)セッション数 / 日次アクティブセッション数。

| 日 | tok/req | out/req | medReq/sess | sessMed cRd/out | worst cRd/out (≥10req、絶対量併記) | hot率 |
|---|---|---|---|---|---|---|
| 07-02 | 136.6k | 620 | 2 | **233x** | 313x(cRd 32.8M), 299x(36.1M), 293x(39.3M) | 30% |
| 07-03 | 104.4k | 687 | 59 | **130x** | 249x(2.5M), 189x(36.8M), 172x(6.7M) | 38% |
| 07-04 | 98.8k | 590 | 59 | **110x** | 891x(5.8M), 479x(15.6M), 283x(40.2M) | 44% |
| 07-05 (部分日) | 127.6k | 1066 | 20 | **83x** | 165x(23.5M), 110x(19.4M), 83x(9.6M) | 38% |

観察:

1. **sessMed cRd/out は 233x→130x→110x→83x と単調減少** [高(数値) / 中(解釈)]。
   同じ期間にセッション形態は「多数の短いセッション(中央値2req)」から
   「少数の長いセッション(中央値59req)」へ移行しており、cache read は文脈長に
   比例して毎リクエスト積み上がるため、**セッションが長くなればこの比は普通は悪化する**。
   長時間化と比率低下が同時に起きているのは、compact による live context の
   切り下げ効果と整合する
2. **hot率は 30%→44% と悪化(改善していない)** [高]。セッション集約の副作用。
   閉ループは「hot になったら切らせる」であって「hot にさせない」ではない
3. worst 比の絶対水準も低下傾向(07-05 の最悪は 165x。07-02 は 313x、07-04 は
   低出力異常値の 891x)。第1報の教訓どおり絶対量(cRd)併読で判断

## 7. 再現手順

```
# セッション内効果
python3 scripts/compaction_effect.py

# 窓合計(クロスチェック)
python3 scripts/usage_report.py 2026-07-02T04:00:00+09:00
```

日次バケット集計はアドホックスクリプトで実施(プラグイン本体には未同梱)。
規則: usage_report.py の main() と同じパース(assistant+usage 行のみ、requestId/
message.id で dedup、`<synthetic>` 除外、subagents/*.jsonl 込み、timestamp 文字列比較)
に加え、(a) timestamp を JST に変換し 04:00 境界で日付バケットへ割当、
(b) compact_boundary レコード(type=system, subtype=compact_boundary)を日次カウント、
(c) セッション単位 rollup から中央値・hot率・worst 比を算出。
Spearman ρ は compaction_effect.py の (preTok, drop%) 29組の平均ランク法による
ランク相関で再計算した。

## 8. 結論

- **セッション内効果(中央値66%削減、フロア~50-64k、単調性)は n=29 でも安定** [高]
- **正規化指標 sessMed cRd/out は稼働後3日で 233x→83x へ単調改善** [高(数値)]。
  セッション長時間化と同時に起きており、compact 効果と整合する [中]
- **cacheRd% とhot率は改善していない** [高]。前者は構造要因で成功指標に不適、
  後者はセッション集約の副作用
- 因果の断定は引き続きしない: n=4日(うち部分日1)・1ユーザー・ワークロード非統制。
  ただし「セッション内の直接測定(66%)」と「正規化系列の単調改善」の2系統が
  同方向を指しており、閉ループが cache read 積み上がりを削っているという説明が
  現時点で最も整合的 [中]
- 測定上の教訓: **測定日当日のバケットを確定値として使わない**(第1報 07-04 行の訂正)
